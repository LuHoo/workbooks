# Workshop IR Schema v1

Canonical overview: `docs/architecture/notebook-generation-and-publication.md`

This document defines the canonical intermediate representation (IR) for
chapter workshop sources under notebooks/support/**/support.Rmd.

Schema identifier: workshop-ir/1.1.0
JSON schema file: scripts/workshop-ir-schema-v1.json

## Goals

- Provide a deterministic, versioned parse artifact.
- Separate parsing from rendering backends.
- Preserve line-level source traceability.
- Keep exporter-specific formatting out of the IR.

## Top-level model

The root object contains:

- schema_version: fixed string workshop-ir/1.1.0
- generated_at_utc: UTC timestamp generated by parser
- source: source file metadata
- chapter: chapter-level metadata
- directives: supported/observed directive names and parsed directive instances
- semantic_references: semantic target registry and extracted semantic reference usages
- chapter_blocks: ordered blocks before first exercise
- exercises: ordered exercise objects

## Exercise model

Each exercise includes:

- exercise_id: stable id (EX-<exercise_ref>)
- semantic_id: stable semantic id (currently equal to exercise_id)
- exercise_ref: source reference such as 5.12
- ordinal: 1-based exercise order in file
- label: heading text without markdown markers
- source_span: start/end and heading line
- blocks: ordered narrative/code blocks

## Block model

Each block includes:

- block_id: stable deterministic id
- semantic_id: stable semantic id (currently equal to block_id)
- block_type: narrative or code
- sequence: 1-based order in chapter or exercise scope
- support_only: true when parsed inside SUPPORT-ONLY marker span
- source_span: start/end and optional code fence lines
- content:
  - narrative block: narrative_lines
  - code block: chunk_header, code_lines, language
- traceability:
  - source_file
  - source_block_key
  - optional workshop_identity_hint for code blocks
- authoring_context:
  - lang_scope (`shared` or language id)
  - mode (`base`, `only`, `override`)
  - kind (`narrative`, `code`, `any`)
  - optional `override_target_block_id`
  - optional requirements list (for example `fsaudit` capability)

## Semantic references in v1.1

The IR exposes first-class semantic reference data through `semantic_references`:

- `token_syntax`: canonical token format string
- `targets`: deterministic semantic target registry with:
  - `semantic_id`
  - `entity_type` (`chapter`, `exercise`, `block`)
  - `entity_id` (the corresponding IR object id)
- `references`: extracted reference occurrences with:
  - stable `reference_id`
  - source location (`source_file`, `source_line`, `source_column`)
  - source ownership (`source_scope`, `source_container_id`, `source_block_id`)
  - `raw_token` and resolved `target_id`

Authoring token syntax:

- `[[ADA:REF target=<semantic-id>]]`

Validation fails loudly when references are unresolved or semantically
inconsistent.

## Directive support in v1

Supported directives in v1.1:

- SUPPORT-ONLY:START
- SUPPORT-ONLY:END
- ADA:BEGIN
- ADA:END
- ADA:REQUIRES

Directives are parser-controlled and marker lines are not emitted as content
blocks. Parsed directive instances are emitted in `directives.instances`.

## Determinism contract

Given identical input and parser version, output is deterministic:

- stable ordering
- stable ids
- stable line spans
- stable block traceability keys

## Error handling contract

The parser fails loudly for malformed input with file and line diagnostics,
including:

- nested SUPPORT-ONLY blocks
- unmatched SUPPORT-ONLY end markers
- unclosed SUPPORT-ONLY block
- unclosed code fence
- duplicate exercise references in one file
- unsupported code fence language for v1 extraction

## Forward compatibility

v1 is intentionally minimal and renderer-neutral.
Future versions may add:

- language-aware directives for Python overrides
- richer execution metadata
- explicit chapter section hierarchy

Versioning rule: backward-incompatible changes require a new schema_version.
